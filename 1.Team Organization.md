# 1. Team Organization and Project Management (การแบ่ง หน้าที่ ในทีม)
Assign roles following the transcendence instructions(one person can have multiple roles if the has 4 members)
"ก่อนที่จะเริ่มโปรเจคจะต้องทำการแบ่งกลุ่มก่อนว่า แต่ละคน ต้องรับหน้าที่อะไรบ้าง(หนึ่งคนสามารถหน้าที่ได้มากกว่า 1 ถ้าเกิดว่า มี คนในกลุ่มสี่คน"

## Product Owner (PO) (คนที่สามารถเข้าใจ painpoint ของ user ได้อย่างทะลุปรุโปร่ง)
Defines the product vision, prioritizes features, and ensures the project meets user needs.
"กำหนด ทิศทางของ ตัวเว็บว่าหน้าตา หรือ การใช้งานๆ ควรออกมาแบบไหน, จัดลำดับก่อน-หลัง ฟีเจอร์ที่ต้องพัฒนาใน เว็บของเรา , และต้องมั่นใจว่า ตอบโจทย์กับสิ่งที่ลูกค้าต้องการจริงๆ

## Project Manager (PM) / Scrum Master (นักจัดการแผนการโปรเจค หรือ นักวิเคราะห์ความเสี่ยง/ Scrum Master(คนที่สามารถดูภาพรวมโปรเจคต้นน้ำยัน ปลายน้ำได้อย่าง แตกฉาน))
Facilitates team coordination and removes obstacles.
"คอยติดตาม คอยให้โปรเจค ไปได้ราบรื่น คอย หลีกเลี่ยงความเสี่ยง ที่อาจจะเกิดขึ้นในโปรเจค"

## Technical Lead / Architect (คนที่ ตีความจาก Product Owner ได้ว่า ลูกค้าต้องอะไร)
Oversees technical decisions and architecture EX. frontend use WordPress?, or other tools, Backend use mysql?, or other tools. 
"เป็นคนที่กำหนด ว่า จะออกแบบระบบ ยังไง เช่น การมีเว็บ เว็บนึง จะต้องมีอะไรบ้าง หน้าบ้าน WordPress ไหม?, หรือใช้เครื่องมือตัวอื่นๆ แล้ว หลังบ้านใช้ mysql ไหม?, หรืือใช้เครื่องมือตัวอื่นๆ"

## Developer (กีกี้ทำตามหัวหน้า)
Implement features and modules
"เป็นคนทำตามหัวหน้าสั่ง พัฒนา ฟีเจอร์เป็นท่อนๆ"


# Team size:

4 people: Some members will have multiple roles (e.g., PM +
Developer, PO + Developer).
"4 คน บางคนอาจจะมี ตำแหน่งควบหลายตำแหน่ง เช่น PM + Developer และ PO + developer"

5 people: Roles can be more specialized, with dedicated PO,
PM, Tech Lead, and 2 Developers.
"5 คน จะแบ่งหน้าที่ได้ชัดเจนกว่า เช่น PO 1  คน, PM 1 คน, Tech Lead 1 คน และ กีกี้ 2 คน"

๒ Recommended Project Management Practices(สื่งที่แนะนำเพื่อให้โปรเจคเสร็จไว)

Regular communication: Meetregularly(weekly or bi-weekly) to sync on progress
and blockers.
"มีทติ้งคุยกันเป็น รายอาทิตย์ ซึ่งจะต้อง สอดคล้องกับ ความคืบหน้าของโปรเจค ที่วางไว้"

Task organization: Use simple tools like GitHub Issues, Trello, or even a shared
document to track who does what
"ใช้เครื่องมือ ที่ต้องติดตามงาน พวก Github Issues หรือ Trello หรือ อะไรก็ตามที่สามารถ ติดตามได้ว่า แต่ละคนไปถึงไหนแล้ว"

Work breakdown: Divide the project into smaller, manageable tasks.
"แบ่งงานให้เล็กๆก่อน จะได้จัดการง่ายๆ"

Code reviews: Try to have at least one other team member review important
code changes.
"ต้องมี อย่างน้อย 1 คนที่จะต้องรีวิว ว่าส่วนสำคัญของโค้ดเปลี่ยนแปลงไปในทางไหนบ้าง"

Documentation: Keep notes of important decisions and how things work.
"เก็๋บไฟล์ หรือ อะไรก็ตามที่บันทึกไว้ว่า ส่วนสำคัญที่ตัดสินใจปรับต่างๆ และ แต่การตัดสินใจปรับฟีเจอร์นั้นๆ ทำงานยังไง"

Communication channel: Use Discord, Slack, or similar for quick team commu
nication
"ช่องทางที่ใช้ในการ มีทติ้งคุยกัน Discord, Slack หรืออะไรก็ตามที่ไว้ใช้คุยกัน"


# Important part
During evaluation, the team will be asked to explain:
  -How roles were distributed. "แบ่งกลุ่มยังไง?"
  -How work was organized and divded "แบ่งงานกันยังไง และ แต่ละงาน จัดการกันยังไง?"
## Member question
  -How you communicated and coordinated as a team "ตัวเราเอง สื่อสาร กับทีมยังไง แล้ว มีบทบาทประสานงานกับทีมแบบไหน"
  -How each member contributed to the project "ตัวเรา เอง ช่วยอะไรบ้างในทีม"

!!!!!All team members must be able to explain the project and their contributions.
"คนในทีมทุกๆคนจะต้อง บอกได้ว่า ช่วยอะไรคนในทีมบ้าง"

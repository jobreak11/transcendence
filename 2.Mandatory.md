# III.2 General requirements
Building an entire project is complicated, and many things can go wrong. To help you, we will provide a list of general requirements that you must follow. If you don't follow them, your project will be rejected.
เหมือนจะบอกโดยนัยว่าโปรเจคมีอะไรหลายอย่างให้ทำเยอะซึ่งพวกเนี่ยจะไกด์ได้ว่าต้องทำอะไรบ้าง ซึ่งถ้าไม่ทำตามนี้ เป็ฯ 0 ทันที

## The requirements are the following():
   -The project must be a web application, and requires a frontend, backend and a database.(โปรเจคต้องเป็นเว็บ ที่มี frontend(หน้าตาเว็บ), backend(ระบบกลางที่ไว้จัดการข้อมูลจากหน้าบ้านสู้ database), ฐานข้อมูล)
  - Git must be used with clear and meaningful commit messages. The repository must show(commit แต่ละรอบต้องเจาะจงชัดเจนว่าทำอะไร ไม่เวิ่นเว้อ ซึ่งรายละเอียดจะตามนี้):
      - Commits from all team members.(commit ที่ต้องมี สมาชิกทุกคนในcommit ทั้งหมด)
      - Clear commit messages describing the changes(ข้อความที่ commit ต้อง อธิบายชัดเจนว่าแก้อะไร)
      - Proper work distribution across the team. (เหมือนน่าจะต้องมีการcommit แล้วบอกต่อด้วยว่าใครจะต้องทำอะไรต่อจากนั้น เพราะข้อนี้เหมือนจะเกี่ยวกับการแบ่งงานให้ เหมาะสม)
   - Deployment must use a containerization solution (Docker, podman, or equivalent) and run with a single command(ตัวเว็บต่างๆโครงของมันต้องจัดอยู่ใน รูปแบบ คอรเทนเนอร์ อารมณ์ เหมือน inception เลย ที่ต้องรันในคำสั่งเดียว)
   - Your website must be compatible with the latest stable version of Google Chrome.(เว็บไซค์ ต้องใช้งานกับ google chrome เวอร์ชั่นล่าสุด ที่เวอร์ชั่นสมบูรณ์แล้ว)
   - No warnings or errors should appear in the browser console.(ต้องไม่มี ข้อความ error หรือ warning เด้งบน browser console)
   - The project must include accessible Privacy Policy and Terms of Service pages with relevant content.(โปรเจคต้องมี Privacy Policy และ Terms of Service)

### Privacy Policy and Terms of Service rules
  - Be easily accessible from the application (e.g., footer links). (เข้าถึงง่ายห้อยท้ายไว้ล่างสุดของเว็บไซค์ก็ได้)
  - Contain relevant and appropriate content for your project.(ข้อมูลต้องเกี่ยวข้องกับหัวข้อ และ เหมาะสม)
  - Not be placeholder or empty pages.(ต้องไม่ใช่ หน้าตัวอย่าง(Template) หรือ หน้าว่างเปล่า)
  ถ้าผิดข้อใดข้อหนึ่ง 0 ทันที

### Multi-user Support rules (Mandatory) (รองรับผู้ใช้ได้พร้อมๆกัน(บังคับ))
  - Multiple users can be logged in and active at the same time.(ผู้ใช้สามารถ ล็อคอิน และ ใช้งาน ได้ในเวลาเดียวกัน)
  - Concurrent action by different users are handled properly.(สิ่งที่ผู้ใช้ทำพร้อมกัน เช่น กดปุ่มจ่ายเงิน บลาๆ อาจมีปัญหาได้ต้องจัดการไม่ให้มี error)
  - Real-time updates are reflected across all connected users when applicable.(เวลามีอัพเดทอะไรใหญ่ในเว็บผู้ใช้ทุกคนต้องรู้โดยทันที)
  - No data corruption or race conditions occur with simultaneous user actions.(ต้องไม่มีการที่ ข้อมูลถูกเขียนทับโดยไม่ตั้งใจ data corruption และ การทำงานต้องเป็นลำดับไม่เกิด race condition)

# III.3 Technical requirements
This section, like the previous one, is mandatory. You will then be able to choose the modules you want to use in the next chapter.ส่วนนี้เหมือนgeneral requirements ที่บังคับเหมือนรอบก่อนๆ ซึ่งสามารถเลือก tools ที่จะใช้ตอนทำ โมดูล

   - A frontend that is clear, responsive, and accessible across all devices.(เว็บหน้าบ้าน ต้องแบ่งหมวดหมู่ชัดเจน, ไม่ช้าอืด, และ ทุกอุปกรณ์ใช้งานเว็บได้)
   - Use a CSS framework styling solution of your choice (e.g., Tailwind CSS, Bootstrap, Material-UI, Styled Components, etc.).(ใช้ framework CSS ตัวอย่างเขามีให้แล้ว)
   - Store credentials (API keys, environment variables, etc.) in a local .env file that is ignored by Git, and provide an .env.example file.(ทำเหมือนinceptionเลยใช้ .env แล้วต้องเป็น git ignore ที่ไฟล์นี้ด้วย และ ต้องใส่ template ของ .env เช่น env.example ด้วย)
   - The database must have a clear schema and well-defined relations(ตัวฐานข้อมูลจะมีตารางโยงกันเป็นความสัมพันธ์ ตัวอย่าสง เช่น ตารางใบบิลจ่ายเงิน ผูกตารางชื่อสินค้า และ ตารางชื่อลูกค้า การออกแบบต้องชัดเจน ไม่ซ้ำซ้อน)
   - Your application must have a basic user management system. Users must be able to sign up and log in securely(ตัวเว็บต้องมีระบบการจัดการuser และ ผู้ใช้สามารถ สมัคร และ ล็อคอิน เข้าสู่ระบบได้อย่างปลอดภัย):
      - At minimum: email and password authentication with proper security (hashed passwords, salted, etc.).(เกณฑ์ขั้นต่ำ มีระบบยืนยันตัวตน ของ email และ รหัสผ่าน ที่ปลอดภัย)
      - Additional authentication methods (OAuth, 2FA, etc.) can be implemented via modules.(ระบบความปลอดภัยเพิ่มเติม เช่น ยืนยัดผ่านการสแกน QR code หรือ สแกนใบหน้า และ อื่นๆ)
   - All forms and user inputs must be properly validated in both the  frontend and backend.(สิ่งที่ผู้ใช้กรอกข้อมูลเข้าไปต้องถูกตรวจสอบทั้ง frontend และ backend อย่างเหมาะสม)
   - Any connection th the backend, from a browser, from a script, from an external API, etc., must be  HTTPS. Connections inside the backend itself (e.g., web server and database, software inside your container(s)) can be without encryption.(การเชื่อมต่อของ backend จากbrowser, จากscript, API และอื่น จะต้องสื่อสารกันในรูปแบบ HTTPS ซึ่งไม่มีการ encryption)

# What is a Framework?
For this project, a framework is defined as a comprehensive tool that provides(สำหรับโปรเจคนี้ ตัว framework ถูกกำหนดเป็นสิ่งที่ครอบคลุมเครื่องมือต่างๆดังนี้):
   - A structured architecture and conventions for organizing code.(โครงร่าง สำหรับการจัดการตัวโค้ด)
   - Built-in features for common tasks (routing, state management, etc.) (เครื่องมือไว้ใช้จัดการงานทั่วๆไป เช่น การ routing และ state management และ อื่นๆ)
   - A complete ecosystem of tools and libraries.(ครอบคลุมไปด้วย เครื่องมือ และ libraries ที่สมบูรณ์)
Examples:(ตัวอย่างเช่น)
   - Frontend frameworks(framwork ของ Frontend): React, Vue, Angular, Svelte, Next.js(พวกนี้เป็น framework)
   - Backend frameworks(framework ของ Backend): Express, Fastify, NestJS, Django, Flask, Ruby on Rails.
   - Not framworks(สื่งที่ไม่ใช่ framwork): jQuery (library), Lodash(utility library), Axios (HTTP client).
Note: React is considered a framework in this context due to its ecosystem and architectural patterns, even though it is technically a library.(React ถือเป็นกรณีพิเศษ ถูกมองว่าเป็น หนึ่งใน framework ในโปรเจคนี้ เพราะ รูปแบบโครงสร้างของมันเอง แม้ว่า โดยหลักทั่วๆไปแล้วจะมองเป็น library ก็ตาม)
